| Category                        | Feature                                | Description                                                                                                                                                  |
| ------------------------------- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Core Architecture**           | Runtime expression engine              | Expressions are defined as strings and parsed at runtime. Expressions can also be constructed manually as ASTs. Compiler-assisted AST generation is planned. |
|                                 | Cached AST                             | Expressions are parsed once and cached; multiple bindings reuse the same AST.                                                                                |
|                                 | Isolated reactive graph per binding    | Each expression binding creates its own isolated reactive graph instance.                                                                                    |
|                                 | Identifier-level reactivity            | Smallest reactive unit is an identifier (field, map key, array index, etc.).                                                                                 |
|                                 | Reference counting                     | Identifiers are reference-counted; when unused, reactive patches/proxies are automatically removed.                                                          |
| **Reactivity Model**            | Model becomes reactive when bound      | Only model properties accessed by the expression become reactive.                                                                                            |
|                                 | Dependency registration during binding | Dependencies are registered when binding the expression to the model (not during normal evaluation).                                                         |
|                                 | Fine-grained tracking                  | Only identifiers accessed by the bound expression are tracked.                                                                                               |
|                                 | Change propagation                     | Identifier mutation → StateManager → dependent expressions re-evaluate.                                                                                      |
|                                 | Independent graphs over same model     | Multiple expressions can bind to the same model without sharing dependency tracking state (identifiers shared via reference counting).                       |
| **Lifecycle & Cleanup**         | Dispose expression                     | Disposing an expression removes its bindings and subscriptions.                                                                                              |
|                                 | Automatic cleanup                      | When no expressions reference an identifier, reactive patches/proxies are removed automatically.                                                             |
|                                 | Memory-safe graph isolation            | Designed to prevent reactive leaks in dynamic runtime systems.                                                                                               |
| **Change Tracking & Debugging** | Expression tree visualization          | Visualize the full expression tree structure (nodes and dependencies).                                                                                       |
|                                 | Live node value inspection             | Inspect each node’s computed value while the model changes.                                                                                                  |
|                                 | ExpressionChangeTrackerManager         | Create change trackers that emit structured change history stacks.                                                                                           |
|                                 | Change history stacks                  | Track old/new values per mutation batch.                                                                                                                     |
|                                 | expression.changeHook                  | Attach custom tracking logic for logging, auditing, or domain-specific tooling.                                                                              |
|                                 | Transaction support                    | Use IExpressionChangeTransactionManager to suspend and resume change emission for atomic update batches.                                                     |
| **Modularity & Composition**    | Composable expressions                 | Expressions can be parsed independently and composed.                                                                                                        |
|                                 | Shared computation                     | Shared expressions evaluate once per dependency change; results reused by dependents.                                                                        |
|                                 | Cached AST reuse                       | Multiple bindings reuse cached parse results while maintaining isolated reactive graphs.                                                                     |
| **Extensibility**               | Pluggable identifier resolution        | Default resolution against bound model; can be customized (e.g., DOM ancestor path or contextual strategies).                                                |
|                                 | Custom hooks                           | Extend behavior via tracker and hook mechanisms without modifying core evaluation logic.                                                                     |
| **Primary Target Use Cases**    | Dynamic rule engines                   | Designed for runtime-editable business logic systems.                                                                                                        |
|                                 | Low-code / no-code systems             | Enables safe user-defined expressions at runtime.                                                                                                            |
|                                 | Reactive calculation engines           | Suitable for pricing engines, financial systems, simulations, and analytics pipelines.                                                                       |
|                                 | Secondary: UI reactivity               | Fully suitable for UI state and dynamic UI logic; equally applicable to rule-based systems.                                                                  |
